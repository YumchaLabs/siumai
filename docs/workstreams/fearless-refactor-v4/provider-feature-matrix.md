# Fearless Refactor V4 - Provider Feature Matrix

Last updated: 2026-03-15

This document is a working matrix for **cross-provider feature alignment**.
It is intentionally pragmatic: it focuses on *what must feel consistent to users* while allowing
provider crates to keep provider-specific implementation details and naming.

## Construction modes (documentation rule)

When examples or docs show more than one construction style, present them in this order:

1. **registry-first** (application code, dynamic routing)
2. **config-first** (provider-specific wiring, tests, integrations)
3. **builder convenience** (compatibility demo, quick start, side-by-side comparison)

## Surfaces

- **Stable (family)**: `siumai::{text,embedding,image,rerank,speech,transcription}` and registry handles.
- **Extension (provider)**: `siumai::provider_ext::<provider>::*` (typed options/tools/metadata/resources).
- **Compatibility**: `siumai::compat::*` and builder-centric convenience entry points.

Alignment work should prioritize **Stable** behavior first, then **Extension** parity where it is
high-value, while keeping **Compatibility** thin and non-authoritative.

For package-boundary decisions (full provider package vs focused package vs compat preset),
see `provider-package-alignment.md`.

For CI ownership and release gating of this matrix, see `validation-matrix.md`.

## Status legend

- `[x]` shipped / aligned
- `[~]` partial / in progress
- `[ ]` not started

For this matrix, an explicitly audited **fail-fast boundary** also counts as aligned. If a
provider intentionally does not support a Stable family but builder / provider / config-first /
registry paths all reject it consistently before transport, mark it `[x]` instead of leaving that
boundary looking unowned.
- `[-]` intentionally provider-specific (extension-only)

## What `aligned` means (acceptance)

A feature is considered aligned when:

1. The **Stable** surface offers a coherent story (API shape + docs + example).
2. Provider-specific differences are isolated under **Extension** or `provider_options_map`.
3. Errors are predictable (same categories, similar messages), and tests cover the invariants.
4. Builder APIs are not required to access the capability (no builder-only features).

## Validation mapping

An aligned row should now have a visible validation owner:

- **provider package compile lane** for provider-owned surfaces
- **facade feature smoke lane** for the corresponding `siumai` feature
- **public example or no-network contract test** for the recommended user path

If a row has architecture notes but no validation owner, it is not considered fully stabilized.

## Next alignment tracks (recommended sequencing)

1. **Tools parity**: tool_choice + tool loop invariants (Stable + provider options).
2. **Structured output parity**: schema path + failure modes + docs (Stable helpers + provider extensions).
3. **Reasoning parity**: enable/budget + extraction semantics + docs (Stable minimal contract; provider-specific details in extension).
4. **Audio family parity**: speech / transcription split, extras boundary, and provider-owned examples.

## Test strategy hooks

When advancing any row above:

- Add a focused, no-network test where possible.
- Add one `construction parity` assertion (registry vs config-first) for the provider.
- Add one `behavioral parity` assertion (same request options produce consistent response shape).
