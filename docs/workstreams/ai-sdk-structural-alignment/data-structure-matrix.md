## Executive summary

The stable prompt/content, usage, request-boundary, and text-completion-family layers are now
close to AI SDK V4 shape parity.

The previous Red structural gap for AI SDK `completionModel()` is now closed on the audited
OpenAI-family paths: Siumai has a stable completion request/response family, core
trait/capability surfaces, registry handles, public facade helpers, audited OpenAI-compatible
`/completions` generate/stream support, and native OpenAI/Azure completion-family execution on the
real provider routes.

The higher-level AI SDK `uploadFile` gap is also much narrower now: Siumai has a public
`siumai::files::upload(...)` helper with explicit input/output structs, automatic media-type
detection, stable `providerReference` results, built-in adapters for the current file-capable
unified/provider clients, and public regression coverage for the main helper semantics.

Siumai now also has a public `siumai::skills::upload(...)` helper aligned with AI SDK
`uploadSkill()` on the currently audited OpenAI and Anthropic paths: explicit skill-file/result
structs, a shared `SkillsCapability` interface mirroring the provider-side `SkillsV4` shape,
provider-owned `skills()` resources, canonical `providerReference` / `providerMetadata`
shaping, Anthropic version-metadata fetch parity, and AI SDK-style OpenAI `displayTitle`
warnings are all present.

Siumai now also has a public AI SDK-style UI-message structural layer:
`siumai::types::{UiMessage, UiMessagePart, UiToolPart, ...}` plus
`siumai::ui::{validate_ui_messages, convert_to_model_messages, convert_to_chat_request}` cover
the static message shape and the `convertToModelMessages`-style conversion path.

What remains in this area is narrower and now clearly scoped: the intentionally absent
`useChat`-style stateful frontend hook layer plus any future provider-owned helper coverage
outside the currently audited file/skill/UI surfaces.

The shared response-side metadata story is also materially cleaner now: `siumai-spec` exposes one
`ProviderMetadataMap` root aligned with AI SDK `ProviderMetadata` (`provider_id -> object`), and
the same shape now backs chat/completion/content/stream/file/skill response structures. The only
intentional exception is UI `providerMetadata`, which still maps onto request-side
`providerOptions` because that is how upstream `convertToModelMessages()` interprets it.

Anthropic custom provider ids are now closer to the audited AI SDK contract too: request parsing
can merge canonical `providerOptions.anthropic` with provider-owned custom roots such as
`my-custom-anthropic`, and top-level non-stream / finish / stream-end `providerMetadata` now
duplicates onto that custom root only when the request actually used it.

Anthropic provider-defined tool/version coverage is now also aligned more closely with the audited
AI SDK package: the shared/provider surfaces include `web_search_20260209`,
`web_fetch_20260209`, `code_execution_20260120`, and `computer_20251124`, Anthropic request
headers inject the same beta tokens as upstream for the new web/computer tools, and the 2026 web
tool path now marks implicit provider-executed `code_execution` calls as `dynamic` on the same
conditions as `hasWebTool20260209WithoutCodeExecution()`.

DeepSeek now follows that same runtime-provider-root rule on the audited chat surface: request
options are read from the resolved provider namespace, response metadata is emitted under that same
root, and the typed DeepSeek helpers now expose keyed accessors for custom-root reads without
guessing arbitrary vendor aliases.

Google/Gemini request shaping is also closer to the upstream provider package now: the audited
`google|vertex` request boundary uses the runtime namespace first for request options and
`thoughtSignature` replay with the sibling namespace as fallback, matching the AI SDK fix for
mixed Google/Vertex payloads.

The helper surface tightened further on the OpenAI-family side as well: OpenAI typed metadata
helpers now expose keyed accessors for explicit provider-root reads, and the built-in Perplexity
compat preset now reshapes response metadata into the same AI SDK-style `images` /
`usage.citationTokens|numSearchQueries` / `cost` layout used by `@ai-sdk/perplexity` instead of
forwarding raw snake_case `usage/images` fragments directly.

The AI SDK first-class-provider gap for DeepInfra is also now closed on the audited public/runtime
paths: Siumai treats `deepinfra` as a built-in unified provider id, reuses the shared
OpenAI-compatible runtime for chat/completion/embedding, keeps provider-owned image generation/edit
routes, and corrects the known DeepInfra usage-accounting drift before it reaches the stable
`Usage` layer.

The AI SDK first-class-provider gap for Vertex MaaS is now also closed on the audited public/runtime
paths: Siumai treats `vertex-maas` as a built-in unified provider id, reuses the shared
OpenAI-compatible runtime for chat/completion/embedding, derives the Vertex `/endpoints/openapi`
base URL from `project + location`, and now accepts Google Bearer auth through token providers or
preexisting `Authorization` headers instead of requiring a fake API key.

The broader Google Vertex unified-builder story is also now structurally closer to AI SDK:
`Siumai::builder().vertex()` and `Siumai::builder().anthropic_vertex()` no longer inject
provider-wide default models, because AI SDK's `createVertex()` / `createVertexAnthropic()`
construct providers and require family-specific `languageModel()` / `embeddingModel()` /
`imageModel()` calls to choose the model explicitly. The provider-owned public facade is now
closer too: `provider_ext::google_vertex::{chat, embedding, image, model_sets}` and
`provider_ext::anthropic_vertex::{chat, model_sets}` expose curated model constants, and the
provider catalog now reuses those same provider-owned curated subsets instead of maintaining
parallel hardcoded lists for `vertex` / `anthropic-vertex`.

The earlier first-class-provider gap for Cohere is now also closed on the audited public/runtime
paths: Siumai treats `cohere` as the canonical native `/v2` provider surface for
chat/embedding/rerank, requires explicit model ids on the unified builder path, and keeps the
OpenAI-compatible Cohere preset only as an opt-in compatibility story instead of the main public
contract.

The earlier first-class-provider gap for Fireworks is now also closed on the audited public/runtime
paths: Siumai treats `fireworks` as a built-in unified provider id, reuses the shared
OpenAI-compatible runtime for chat/completion/embedding/transcription, and now routes image
generation/edit through provider-owned Fireworks workflow and `image_generation` endpoints with
default image/edit fallback models plus contract/public-path coverage.

The earlier TogetherAI runtime split is now also structurally closer to the audited package
boundary: canonical `togetherai` keeps chat/completion/embedding/speech/transcription on the
shared OpenAI-compatible runtime, but image generation/edit and reranking now stay provider-owned
on the unified provider surface. Public typed `TogetherAiImageOptions` plus provider-owned image
regression coverage now pin the `/images/generations` + `image_url` edit contract instead of
reusing the generic OpenAI-compatible image lane.

Stable provider typing is also now structurally closer to the audited AI SDK provider package set:
`ProviderType::{Azure, Cohere, TogetherAi, Bedrock, Mistral, Fireworks, Perplexity}` now exist
alongside the previously-promoted `DeepInfra` and Google Vertex variants, so provider catalog,
retry defaults, validator heuristics, and client metadata no longer degrade those audited provider
packages to `Custom(...)`. The important nuance is now provider-specific: `mistral` and
`perplexity` still reuse the shared OpenAI-compatible runtime internally, while `fireworks` now
uses that shared runtime only for the text/audio families and keeps provider-owned image routing at
the unified provider layer.

The package-owned compat surface for `mistral` and `perplexity` is also now closer to the audited
AI SDK provider packages themselves: Mistral now has public typed language-model options
(`safePrompt`, document limits, `structuredOutputs`, `parallelToolCalls`, `reasoningEffort`)
instead of forcing raw `providerOptions.mistral` JSON, Perplexity typed options now also keep an
AI SDK-style camelCase public surface while explicit compat normalization lowers them onto
Perplexity's snake_case wire fields only at transport time, and both Mistral plus Perplexity now
reuse provider-owned curated model constants/defaults so the public catalog no longer advertises
stale Perplexity legacy ids.

That Mistral audit is now also documented explicitly as its own package workstream:
`docs/workstreams/mistral-package-surface-alignment/` records the audited `chat + embedding`
boundary from `@ai-sdk/mistral`, the intentional lack of completion/image support, and the split
between public typed options and the lower wire contract.

The dedicated MoonshotAI compat wrapper is also now closer to the audited `@ai-sdk/moonshotai`
package boundary: canonical public/runtime identity is `moonshotai`, the old `moonshot` id is
kept only as a hidden migration alias, typed `thinking` / `reasoningHistory` request options now
normalize onto the Moonshot wire keys, curated Kimi model constants/defaults follow the audited
package subset, and completion/image/embedding stay intentionally unsupported because the upstream
package itself is chat-only.

One architectural point is now explicit in the workstream baseline: AI SDK does have a shared
image abstraction through `ProviderV4.imageModel(...)`, `ImageModelV4`, and `generateImage(...)`,
so Siumai should absolutely preserve one stable image request/result surface. But that shared layer
does not imply a single generic runtime implementation. The audited upstream provider packages keep
hybrid/provider-specific image execution for Fireworks, DeepInfra, TogetherAI, Google Vertex,
OpenAI, and others, so Siumai should treat "shared interface, provider-owned runtime" as the
default image architecture unless upstream actually consolidates those execution paths.

The remaining structural risk is concentrated in streaming:

- the stable stream-part shape exists
- the runtime transport has a first-class `ChatStreamEvent::Part(ChatStreamPart)` channel
- but parser/provider coverage is still incomplete
- and some protocol-native replay fields still live outside the stable model

The main raw-chunk parity gap on the audited text/completion paths is now closed: stable request
types now have a runtime-only `StreamRequestOptions.include_raw_chunks` lane, both
`siumai::text::StreamOptions` and `siumai::completion::StreamOptions` map to that lane instead of
polluting `providerOptions`, the audited OpenAI-compatible plus Anthropic chat parsers emit stable
`raw` parts in the same relative order as AI SDK (`stream-start -> raw -> response-metadata -> ...`
on the first chunk), native Bedrock Converse JSON streaming now also emits the upstream-style
Bedrock preamble (`stream-start -> response-metadata -> raw -> ...`) with stable `error` parts for
provider error envelopes, stable `text-*` / `reasoning-*` / `tool-input-*` / `tool-call` /
`finish` parts, Bedrock cache-aware usage/provider metadata on `finish`, streamed
`stopSequence` / `raw_finish_reason` retention on the final response, and response-side reasoning
metadata plus Mistral tool-call-id normalization on the non-stream lane. The audited
OpenAI-compatible plus native OpenAI/Azure completion SSE paths now also emit the same stable
lifecycle ordering on the completion lane. First-chunk parse failures are now also
narrower on the audited AI SDK lanes: OpenAI-compatible, Anthropic, native OpenAI completion, and
native Cohere keep `stream-start` before optional `raw` and the parse error, native Bedrock now
keeps its first-chunk preamble before optional `raw` and the parse error, and Gemini now keeps
`stream-start` before the parse error instead of skipping the lifecycle start entirely.

The audited Bedrock request-boundary story is now also structurally closer to the AI SDK package:
`provider_ext::bedrock` now exposes typed `reasoningConfig`, `serviceTier`, and file-part
`citations` helpers instead of relying on raw `providerOptions.bedrock` JSON, unknown top-level
Bedrock passthrough fields are preserved like upstream instead of being dropped during
normalization, Anthropic Bedrock requests now derive `additionalModelResponseFieldPaths`,
`thinking`, `anthropic_beta`, and the same `maxReasoningEffort` routing split that upstream uses
(`output_config.effort`, `reasoning_effort`, or nested `reasoningConfig`), and Anthropic
structured JSON output now uses native `output_config.format` on the same audited Bedrock routes
where the AI SDK avoids the reserved JSON tool. The audited prompt converter now also keeps
message-level `cachePoint`, maps user `file` parts onto Bedrock `document` / `image` blocks with
upstream-style filename stripping plus typed document citations, replays Bedrock reasoning
`signature` / `redactedData` from canonical request-side `providerOptions`, supports tool-result
`text` + `image-data` content, and normalizes request-side Mistral tool ids on both tool calls and
tool results. On the response/public side, Bedrock reasoning parts now also have a typed metadata
view plus replay helper: `BedrockContentPartExt::bedrock_reasoning_metadata()` exposes
`signature` / `redactedData`, and `assistant_message_with_reasoning_metadata(...)` moves those
response-side fields back onto next-turn request-side `providerOptions.bedrock`.

The biggest remaining gaps are:

1. parser/provider stream-part emission is still uneven outside the already-audited
   OpenAI Responses / Anthropic / Gemini main paths plus the audited OpenAI-family completion lane
2. the runtime stream story is still split across `ChatStreamEvent`, `ChatStreamPart`, and the
   compatibility overlay even though the public naming gap is now reduced by the
   `LanguageModelV4*` aliases
3. shared non-chat request abstractions are now structurally close to the audited AI SDK shape:
   `ImageEditRequest` models typed multi-input `images[]` + `mask` semantics, and
   `ImageVariationRequest` also carries a typed file/url image input instead of a raw byte bag.
   Those typed image inputs now also expose per-input `providerOptions`, bringing the shared image
   input lane closer to AI SDK `ImageModelV4File`. The shared image request family exposes
   top-level `aspectRatio` plus shared `seed` across generation/edit/variation, compat/OpenAI
   image warning semantics now match the AI SDK-style `unsupported { feature }` lane for top-level
   `aspectRatio` / `seed`, xAI / Google / Vertex supported image paths consume those canonical
   shared fields directly, and the shared executor materializes `data:` / `http(s):` URL-backed
   edit/variation inputs before synchronous multipart/inline transformers run on the audited
   OpenAI/OpenAI-compatible/Vertex paths. The remaining follow-up work in this area is no longer
   the shared request shape itself; it is narrower provider/runtime coverage and provider-owned
   option parity outside that stable structural lane
4. the OpenAI-compatible provider-settings surface is now materially closer to AI SDK:
   `metadataExtractor`, `includeUsage`, `transformRequestBody`, `queryParams`, and an explicit
   `supportsStructuredOutputs` provider-level policy all exist on the public config/builder/runtime
   path now, and the default `supportsStructuredOutputs` behavior now also matches AI SDK's
   conservative default `false`
5. the remaining high-level AI SDK helper gaps are now concentrated above the core runtime:
   Rust now has both `uploadFile` and `uploadSkill` high-level helpers, and `uploadSkill`
   additionally has a shared `SkillsCapability` plus audited OpenAI/Anthropic coverage matching
   the current AI SDK provider scope. Rust now also has a public AI SDK-style
   `UIMessage` / `convertToModelMessages` layer, so the remaining work is the final typed/pass-through
   story for provider-owned file-upload options plus the intentionally absent
   `useChat`-style stateful frontend hook layer

## What is still not structurally complete

### 1. Protocol-only replay hints now have an explicit runtime carrier where they are truly protocol-only

OpenAI Responses `rawItem` / `outputIndex` replay is now modeled through a separate runtime replay
carrier attached alongside stable `ChatStreamPart` semantics.

That means:

- `ChatStreamPart` stays semantic
- protocol replay state is not stuffed into generic `providerMetadata`
- same-protocol replay no longer depends on loose provider-scoped custom JSON payloads

Current conclusion after auditing Anthropic against
`repo-ref/ai/packages/anthropic/src/anthropic-messages-language-model.ts`:

- Anthropic reasoning `signature` and `redactedData` are not protocol-only replay state
- AI SDK exposes them on stable `reasoning-*` part `providerMetadata`
- Siumai now follows that same rule

So the dedicated runtime replay carrier remains the right tool for true wire-only fidelity such as
OpenAI Responses `rawItem` / `outputIndex`, but it is no longer the preferred home for Anthropic
reasoning signatures.

### 2. The runtime stream story is structurally correct but still split

Today Siumai has three relevant layers:

- `ChatStreamEvent`
- `ChatStreamPart`
- `LanguageModelV3StreamPart` / `LanguageModelV4StreamPart`

This is workable, but still more complicated than the AI SDK union story.

Recommendation:

- keep `ChatStreamEvent` as the transport/runtime envelope
- keep `ChatStreamPart` as the primary semantic runtime part
- keep the new `LanguageModelV4*` aliases as the recommended public naming for the upgraded
  overlay
- keep `LanguageModelV3StreamPart` as a compatibility name, not as the long-term semantic center

### 3. Parser migration is still uneven

The runtime part channel is only fully useful once parsers emit it directly when they already know
the stable meaning.

Current priority order:

1. tighten the remaining transport-hint raw edges and the analogous parse-failure cases outside the now-audited OpenAI-compatible, Anthropic, Gemini, native OpenAI completion, native Cohere, and native Bedrock first-chunk stream lanes; after re-checking `repo-ref/ai/packages/amazon-bedrock/src/bedrock-chat-language-model.ts`, Bedrock is no longer carrying a known `source` / `custom` stream-part gap because the current upstream `doStream()` contract itself only exposes `stream-start`, `response-metadata`, `raw`, `text-*`, `reasoning-*`, `tool-*`, `finish`, and `error`

3. keep auditing provider/runtime coverage around the now-stabilized shared non-chat surface, especially residual provider-owned video knobs
4. keep moving public tests and examples toward `event: part` / stable-part-first assertions so the explicit Axum `{ part, replay }` export contract becomes the default downstream expectation

### 4. Extras orchestration still trails AI SDK in a few high-level controls

The high-level extras step/result surface is now much closer to AI SDK:

- stable `call_id`
- stable `model { provider, model_id }`
- unified `content`
- step-scoped `request` / `response`
- telemetry `function_id` / `metadata`
- stable `raw_finish_reason` backed by `ChatResponse.raw_finish_reason`
- canonical user-defined `context` flowing through `prepare_step`, tool execution, `StepResult`,
  and `on_finish`

The main remaining gaps on this row are now narrower:

- extras still centers the lower-level `messages`, `tool_calls`, and `tool_results` contract even
  though standardized `tool_call_views()` / `tool_result_views()` and static/dynamic splits now
  exist; AI SDK still goes further with stronger type-level tool families
- tool-result projections now resolve `input`, but the stable `ContentPart::ToolResult` itself
  still does not store the full AI SDK typed/dynamic payload directly
- the raw-finish chain is now structurally present on the audited paths because
  `ChatResponse.raw_finish_reason` exists as the stable carrier and extras `StepResult` forwards
  it directly; the remaining work is broader provider coverage review, not missing shared structure

Recommendation:

- keep the newly landed `context`, per-step model override, and normalized tool projections as the
  stable baseline
- continue the next fearless-refactor cut on moving more of those normalized tool semantics into
  the stable content layer before revisiting deeper provider-native finish-reason plumbing
